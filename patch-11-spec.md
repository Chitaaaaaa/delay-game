# Patch 11：KPI松紧背景系统 + IP授权方系统

---

## 概述

两个背景层系统，互相咬合，共同构成玩家头顶的外部压力环境：
- **KPI松紧**：决定老板干预频率和事件池难度
- **IP授权方**：决定质量地板和砍功能的自由度

---

## 新增游戏状态字段

```javascript
kpiState: 'loose' | 'normal' | 'tight',   // 默认 'normal'
ipType: 'strong' | 'normal' | 'none',      // 开局派生
ipActive: boolean,                          // IP方是否存在（none时为false）
```

---

## 一、KPI松紧系统

### 状态定义

| 状态 | 体感 | 事件池影响 |
|------|------|-----------|
| loose（松） | 老板满意，创作空间大 | 增加权重+30%：团队自驱、创意提案、意外惊喜；减少权重-30%：管理层干预、需求变更 |
| normal（正常） | 默认基准 | 无调整 |
| tight（紧） | 老板频繁出现，压力持续 | 增加权重+30%：管理层check-in、需求反复、加班压力、对赌追责；减少权重-30%：创意类事件 |

### 触发路径

**路径一：对赌协议联动**
```
月底milestone评估时触发对赌检查：
  若本月进度增量 < 目标增量（每月约需+16%）：
    kpiState 收紧一档（loose→normal，normal→tight）
    触发对赌追责事件（见事件文案）

  若连续2个月超额完成：
    kpiState 缓解一档（tight→normal，normal→loose）
```

**路径二：市场热潮联动**
```
市场热潮事件（每局随机触发1-2次，月2-5之间）：
  玩家选「坚守方向」→ kpiState 收紧一档
  玩家选「表面融入」→ kpiState 维持不变
  玩家选「全力跟风」→ kpiState 暂时松一档，但 gameDirection 受损
                       （方向匹配度降级，影响最终结局判定）
```

**主动缓解**
```
向上管理行动（2AP）：
  累计执行3次后，下一个milestone评估时kpiState有50%概率缓解一档

milestone评估超预期（进度增量超目标20%以上）：
  自动缓解一档
```

### KPI状态玩家感知方式

不显示KPI状态标签，玩家通过事件频率和文案语气感知：
- tight状态：老板事件文案语气变短、变硬
- loose状态：老板事件减少，偶尔出现正面旁白

## 二、IP授权方系统

### 开局派生规则

由"你接手的项目"开局卡决定IP方出现概率，出现后随机分配强度：

| 项目卡 | IP方出现概率 | 强度分布（若出现） |
|-------|------------|-----------------|
| 全新立项 | 30% | 强势20% / 普通80% |
| 中途接盘 | 70% | 强势40% / 普通60% |
| 秽土转生 | 90% | 强势60% / 普通40% |

出现时开局揭示IP方存在（可见），强度不揭示（不可见，靠事件感知）。

### IP方三种事件类型

**① 保护型（强势IP专属）**

触发条件：资源掠夺类事件发生时（管理层抽调人员/预算）

```
效果：该次资源掠夺事件强制出现第三选项「IP方介入」
      选择后：资源掠夺取消，但IP方关系消耗一次（每局限用2次）

文案：
「你的手机震了一下。是IP方的法务。他们发来一封邮件，
 措辞非常礼貌，意思非常明确：这个项目的资源不能动。
 你第一次感谢一份律师函。」
```

**② 施压型（强势+普通均有）**

触发条件：月底milestone时，qualityDebt >= 50

```
强势IP方：
  强制执行品质对齐
  qualityDebt -= 20，progress -= 8
  无选项，直接结算

  文案：
  「IP方发来本月品质审查报告，标注了17处与IP调性不符的内容。
   没有商量余地。你让团队停下来，把这些地方重新做了一遍。」

普通IP方：
  触发提醒事件，有选项

  文案：「IP方说，他们对这个月的进展有些担心。」

  A：「我们会注意的。」→ 无效果（敷衍过去）
  B：主动对齐品质 → qualityDebt-10，progress-4（主动承担，损失小一点）
  C：「请相信我们的专业判断。」→ 信任度-1（IP方不高兴了）
```

**③ 枷锁型（强势IP专属）**

触发条件：玩家执行「砍功能」行动时

```
qualityDebt >= 60：
  硬锁——行动失效，弹出警告
  文案：「IP方合同第14条：核心内容不可删减。
         你看了一遍合同，确实写着。」

qualityDebt 40-59：
  软警告——可以执行，但附带代价
  信任度-1，qualityDebt额外+5
  文案：「你砍掉了那个功能。IP方没说什么，但你知道他们看见了。」

qualityDebt < 40：
  正常执行，无任何限制
```

## 三、KPI × IP咬合场景

以下是两个系统同时生效时的典型局面，程序侧无需特殊处理，自然叠加即可：

```
场景：KPI tight + 强势IP方存在

玩家处境：
  - 老板频繁出现，需要快速推进进度
  - 想砍功能但qualityDebt已经50+，IP方会踩
  - 想全员冲刺但qualityDebt继续涨，下个月IP方施压更重

这是本游戏最高压的局面，两面夹击，没有免费出口。
```

## 接入说明

- KPI事件池权重调整：在现有事件抽取逻辑中，读取 `kpiState` 对对应事件类型的权重做±30%修正
- IP派生：在开局卡牌结算后执行，结果写入 `ipType` 和 `ipActive`
- IP枷锁：在「砍功能」行动执行前插入条件检查
- IP施压：接入月底milestone结算流，在质量检查之后执行
- IP保护：在资源掠夺类事件的选项生成时，检查 `ipType === 'strong'` 且剩余保护次数 > 0
