# Patch 39：意外事件系统

> 前置：Patch 01-38 已完成
> 目标：定义6种意外事件的触发逻辑、归因机制、效果
>       意外事件不由玩家选择触发，是世界对项目状态的反应

---

## 一、意外事件 vs 常规事件

| 维度 | 常规角色事件 | 意外事件 |
|------|------------|---------|
| 触发方式 | 角色本周「出现」→ 带来问题 | 游戏状态达到阈值 → 自动触发 |
| 有无归因 | 有（该角色） | 大部分有，由亲和性算法选出责任人 |
| 关系值结算 | 有 | 视事件类型（见各条目）|
| 是否必须处理 | 是（本周事件）| 是 |

---

## 二、6种意外事件

### 2.1 灵机一动

**触发条件**：`directionClarity < 50` 且 `week <= 12`，每4周最多触发1次

**归因**：亲和性选最高的 幻想家 > 远见（如果两个都不在场，由「某个匿名同事」带来）

**事件模板**：

```
情境：[角色名] 发来一条消息……
对话：「我刚试完[竞品/新游戏]，有个想法——我们能不能加一个[功能X]？
      应该不难，而且我觉得玩家一定会喜欢的。」
```

| 选项 | 效果 | 关系变化 |
|------|------|---------|
| 「先记下来，等有余力再说」| directionClarity-3（这个想法在脑子里转）| 归因角色 +0 |
| 「可以，下周排进去看看」| directionClarity-8, progress-3 | 归因角色 +1 |
| 「现在不是时候，砍掉」| directionClarity+2 | 归因角色 -1 |

---

### 2.2 救火

**触发条件**：另一项目进入发布期（随机在第8~16周触发1次），与 studioScale 无关

**归因**：由 Tom / 老板通知，无角色关系结算

**事件模板**：

```
情境：Tom 打来电话：
对话：「[隔壁项目名]下个月上线，美术资源不够，
      我需要你这边支援一半的人过去，大概两个月。」
```

| 选项 | 效果 |
|------|------|
| 「没问题，支持公司」| progress-8/周（持续2周），morale-5, bossTrust+2 |
| 「能不能少一点，我这边也很紧」| progress-4/周（持续2周），morale-2, bossTrust+0 |
| 「我们现在真的不行」| progress不变, bossTrust-2, morale+3（团队感到被保护）|

---

### 2.3 减员

**触发条件**：任意角色 relationship ≤ -2，或随机（每局至少1次，week >= 8）

**归因**：relationship 最低的在场角色，或随机选一个在场角色

**事件模板**：

```
情境：[角色名] 约你喝咖啡……
对话：「[待定——根据关系值高低有不同版本]
      我……最近在看一些机会。」
```

**三种结果**：

| 玩家选择 | 条件 | 效果 |
|---------|------|------|
| 挽留（给条件）| — | budget-8, 该角色relationship+1，morale+2（团队看到你在保人）|
| 好聚好散 | — | projectHeadcount-1，morale+3（团队透气），该角色离场 |
| 拖住（画饼）| — | 该角色本周留下，关系值-1，2周后再次触发本事件 |

该角色离场后：`characters[i].active = false`，不再出现。

---

### 2.4 甩锅

**触发条件**：某个事件（任意类型）的结果为负面，同周内关系值 ≤ -1 的角色存在

**归因**：关系值最低的在场角色

**事件模板**：

```
情境：出了事之后……
对话：「[角色名] 在群里发了一条消息：
      『这个问题之前我提过，当时没有得到重视……』」
```

| 选项 | 效果 | 关系变化 |
|------|------|---------|
| 在群里回应，澄清责任 | morale-2（撕逼），bossTrust-1 | 归因角色 -1 |
| 私下找他谈 | morale+1（处理掉了），progress-1（时间成本）| 归因角色 +0 |
| 不回应，让它过去 | morale-3（大家都看着）| 归因角色 -1 |

---

### 2.5 屎山

**触发条件**：`qualityDebt >= 4`，每3周最多触发1次

**归因**：完美主义 > 远见（亲和性最高者）

**事件模板**：

```
情境：[角色名] 来找你：
对话：「我需要跟你说一件事。
      新来的同学看了一下[模块X]的代码……
      他说，他不知道从哪里开始改。」
```

| 选项 | 效果 | 关系变化 |
|------|------|---------|
| 「专项整治，这周先停新功能」| progress-10, qualityDebt-3 | 归因角色 +1 |
| 「边做边改，能跑就行」| progress-3, qualityDebt-1 | 归因角色 +0 |
| 「新人适应一下就好了」| qualityDebt+1, morale-3（他们听到了）| 归因角色 -1 |

---

### 2.6 雷公

**触发条件**：随机（每局约1次，week 6~18 随机），无状态阈值触发

**归因**：无（纯外部事件，无角色关系结算）

**事件模板**：

```
情境：周三下午，突然——
对话：「停电了。

      不是短路，是整栋楼的电力线路超负荷。物业说，
      今天剩下的时间无法恢复。

      你的笔记本还有两小时电。」
```

| 选项 | 效果 |
|------|------|
| 让大家回家远程，今天继续 | progress-2, morale+3（意外假期）|
| 去咖啡馆继续工作，死磕 | progress-1, morale-2（很累）|
| 就停工吧，明天补回来 | progress-4, morale+5 |

## 三、背景特殊选项

意外事件中，5种有归因的事件（雷公除外）各有1~2个背景特殊选项：

| 意外事件 | 触发背景 | 特殊选项文本 | 效果 |
|---------|---------|------------|------|
| 灵机一动 | designer | 「我来评估一下这个想法的范围和优先级」| directionClarity只-3，可以中性处理不伤关系 |
| 灵机一动 | pm | 「先算一下加这个功能的ROI」| 若结果差 → directionClarity+2（砍掉有依据），relationship-0 |
| 救火 | pm | 「能不能把支援时间压到6周，我来谈」| progress损耗减1周，bossTrust不变 |
| 减员 | artist | 「我理解，文化氛围的问题，我早看出来了」| 挽留时relationship+1额外加成；放走时morale+5（团队感到被理解）|
| 减员 | education | 「我们先聊聊，你真正想要的是什么」| 画饼选项改为真实留人可能，relationship+1 |
| 甩锅 | pm | 「我来处理这个，不要在群里撕」| 化解morale损耗，relationship归因角色不变但影响可控 |
| 甩锅 | engineer | 「我看了一下日志，责任不在这里，我来说明」| 可以技术性澄清，bossTrust+1（老板看到你处理专业）|
| 屎山 | engineer | 「我看看代码，告诉你真实的重构成本」| 更精准的qualityDebt评估，可以协商部分重构而非全部 |

雷公无背景特殊选项（外部事件，背景无助于应对）。

---

## 四、触发优先级与频率控制

- 同一周只触发1个意外事件
- 意外事件与常规角色事件互斥（同周只出现一种）
- 频率上限：每3周最多1次意外事件（雷公除外，雷公强制插入）
- 甩锅是例外：可以附加在常规事件结果之后，不占用本周事件名额

- [ ] 雷公：无归因角色，关系值不结算
