## 改动一：周内流程拆成两个阶段

### 当前流程（错误）
```
事件出现 → [工作模式 + 行动菜单 + 事件选项] 混在一起
```

### 新流程
```
阶段一：主动（周初规划）
  选工作模式 → 花AP做行动 → 点「进入下半周」确认

阶段二：被动（突发响应）
  意外事件出现 → 选A/B/C → 看结果 → 下一周
```

### 实现要点

1. 新增 `weekPhase` 状态：`"planning"` | `"event"`，初始值 `"planning"`

2. **planning 阶段**显示：
   - 工作模式选择器（现有 `WorkModeSelector`）
   - 行动菜单（现有 `ActionMenu`）
   - 确认按钮：「本周安排完毕，看看发生了什么 →」

3. 确认按钮点击后：
   - `weekPhase` 切换到 `"event"`
   - 此时才渲染事件卡片和选项

4. **event 阶段**只显示：
   - 事件人物卡 + 对话框
   - A/B/C 选项（不再显示 WorkModeSelector 和 ActionMenu）

5. 选完选项、看完结果、点「下一位来访者」时：
   - `weekPhase` 重置回 `"planning"`
   - AP 重置为 0
   - 推进到下一周

## 改动二：未消耗AP的丢弃提示

在 planning 阶段的确认按钮上方，当 `apLeft > 0` 时显示一行提示：

```
你还有 X AP 未使用，确认后将消失。
```

样式：fontSize 10，color `#555`，fontFamily monospace，字体偏暗，不刺眼，只是一个事实陈述。

AP 丢弃不做任何补偿，直接归零。

## 改动三：月度骨架

### 月底强制 milestone 事件

第 4、8、12、16、20 周末（即每月最后一周处理完事件后），在「下一位来访者」之前插入一个**月度小结卡**。

第 24 周是游戏终局，不需要月度小结。

### 月度小结卡内容

显示一张独立的卡片（替代普通事件卡），内容：

```
第X月结束

本月进度推进了 [delta]%
距离上线还有 [100 - progress]%
剩余 [weeksLeft] 周

[一行月份专属的氛围文字]
```

月份专属氛围文字（按月份写死）：
- 月1结束：「原型期过去了。方向定了吗？」
- 月2结束：「两个月了。团队开始知道彼此的脾气了。」
- 月3结束：「过半了。但进度过半了吗？」
- 月4结束：「四个月。你开始能感觉到什么东西在收紧。」
- 月5结束：「最后一个月到了。」

只有一个按钮：「进入第X+1月 →」，点击后正常推进到下一周。

### 月度小结的触发逻辑

在 `nextEvent`（点击「下一位来访者」）的逻辑里：

- 推进周数后，判断 `newWeek % 4 === 1`（即刚进入新月的第一周）
- 且 `newWeek <= 21`（第24周之前）
- 则先显示月度小结卡，而不是直接加载下一个事件

用新状态 `showMonthSummary: bool` + `monthSummaryData: { month, delta, ... }` 控制。

### 月份专属行动解锁（最简版，先占位）

在 `ACTIONS` 数组的 `available` 函数里，现有的按周数判断逻辑已经覆盖了这个需求（如 `freeze` 要求 week 9-16，`cut`/`sprint` 要求 week 17+）。

这次不新增专属行动，只确保现有的按月解锁逻辑正确显示在 planning 阶段（而不是 event 阶段）。
